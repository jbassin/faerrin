# MR Response Implementation Plan

## Overview

Enable stakeholders to review GitLab MR comments and have their responses applied by the LLM. Two response types:
1. **Tracked discussions** (speculative/contradiction notes posted by the system): stakeholder replies "approve" or "deny" in the thread; approved claims get a concrete edit committed to the MR branch.
2. **Diff discussions** (inline comments on changed files): stakeholder leaves a free-form instruction (e.g. "rename to X.md", "delete this"); LLM interprets and commits the change.

A new `bun run respond <name>` command kicks off the process.

---

## Current State Analysis

- `addNote()` posts to `/merge_requests/:iid/notes` — flat top-level notes with no threading support.
- Note IDs are discarded after posting; there is no mapping from GitLab note → proposal index.
- `GitLabClient` (`src/gitlab/client.ts`) has six write-only methods — no read endpoints at all.
- `CommentProposal` has `{ message, reason, relatedPath, citations }` but no concrete edit content; approving one requires a new LLM call.
- The ledger (`src/transcript/ledger.ts`) stores `prUrl` but not `mrIid`, which is required for all MR API calls.
- No `state/submissions/` directory exists.

## Desired End State

After `bun run respond <transcript-name>`:
- All "approve" replies in tracked discussion threads produce commits on the existing MR branch.
- All "deny" replies receive a "Denied — no changes applied." bot reply with no commit.
- All unhandled diff discussions receive LLM-interpreted commits.
- Every processed discussion gets a bot reply ("Applied." or "Denied.") so re-runs skip it.

Verify by: running `bun run respond` twice in a row — the second run produces no commits and no new replies.

### Key Discoveries
- `src/gitlab/submit.ts:101-104` — the loop calling `addNote` per comment proposal discards the return value; switching to `createDiscussion` lets us capture the discussion ID.
- `src/gitlab/apply.ts:8` — `buildCommitActions(proposals, ctx)` accepts any `Proposal[]` and reads files from disk; we can feed it proposals generated by the new LLM call without touching `apply.ts`.
- `src/reconcile/propose.ts:74-96` — the proposer LLM prompt and Zod schema pattern; the speculative-approval LLM call mirrors this exactly.
- `src/transcript/ledger.ts:33-43` — `LedgerEntrySchema` with optional `prUrl`; adding optional `mrIid` follows the same pattern.

## What We're NOT Doing

- No webhook/event-driven triggers — polling only (user runs the command manually).
- No structured comment syntax beyond the "approve"/"deny" prefix — diff comment instructions are free-form English.
- No partial approval — a stakeholder either approves or denies the whole speculative comment.
- No re-submission of denied speculative comments.
- No new ledger stage — `respond` is ad-hoc and doesn't gate subsequent pipeline stages.
- No changes to the existing segment → extract → resolve → match → propose pipeline.

## Implementation Approach

Five incremental phases. Phases 1–3 change the submit side (switching from notes to discussions and tracking state). Phases 4–5 add the respond capability. Each phase leaves tests green.

```typescript
import { rename } from 'node:fs/promises';
import { z } from 'zod';

export const DiscussionMappingSchema = z.object({
  discussionId:  z.string(),
  proposalIndex: z.number().int().nonnegative(),
});

export const SubmissionsFileSchema = z.object({
  filename:    z.string(),
  mrIid:       z.number().int().positive(),
  discussions: z.array(DiscussionMappingSchema),
});

export type DiscussionMapping = z.infer<typeof DiscussionMappingSchema>;
export type SubmissionsFile   = z.infer<typeof SubmissionsFileSchema>;

export async function readSubmissions(path: string): Promise<SubmissionsFile | null> {
  const file = Bun.file(path);
  if (!(await file.exists())) return null;
  return SubmissionsFileSchema.parse(JSON.parse(await file.text()));
}

export async function writeSubmissions(path: string, data: SubmissionsFile): Promise<void> {
  const tmp = `${path}.tmp`;
  await Bun.write(tmp, JSON.stringify(data, null, 2) + '\n');
  await rename(tmp, path);
}
```

## Phase 2: GitLab Client Read/Write Additions

### Overview

Add `createDiscussion`, `listDiscussions`, and `addDiscussionNote` to the client. Extend `CommitAction` with `delete` and `move` action types (needed by the respond phase to rename or delete files).

### Changes Required

#### 1. New types and extended `CommitAction`
**File**: `src/gitlab/client.ts`

```typescript
// New read types
export interface DiscussionNote {
  id:       number;
  body:     string;
  author:   { username: string };
  position: {
    position_type: string;
    new_path:      string;
    new_line:      number | null;
    old_line:      number | null;
  } | null;
}

export interface Discussion {
  id:              string;
  individual_note: boolean;
  notes:           DiscussionNote[];
}

// Extended CommitAction (replace existing interface)
export type CommitAction =
  | { action: 'create' | 'update'; filePath: string; content: string }
  | { action: 'delete'; filePath: string }
  | { action: 'move';   filePath: string; previousPath: string; content?: string };
```

#### 2. New client interface methods
**File**: `src/gitlab/client.ts` — add to `GitLabClient` interface:

```typescript
createDiscussion(mrIid: number, body: string): Promise<{ discussionId: string }>;
listDiscussions(mrIid: number): Promise<Discussion[]>;
addDiscussionNote(mrIid: number, discussionId: string, body: string): Promise<void>;
```

#### 3. Implementations

```typescript
async createDiscussion(mrIid, body) {
  const data = await request(
    'POST', `/merge_requests/${mrIid}/discussions`, { body }
  ) as { id: string };
  return { discussionId: data.id };
},

async listDiscussions(mrIid) {
  const results: Discussion[] = [];
  let page = 1;
  while (true) {
    const batch = await request(
      'GET', `/merge_requests/${mrIid}/discussions?per_page=100&page=${page}`
    ) as Discussion[];
    results.push(...batch);
    if (batch.length < 100) break;
    page++;
  }
  return results;
},

async addDiscussionNote(mrIid, discussionId, body) {
  await request(
    'POST',
    `/merge_requests/${mrIid}/discussions/${encodeURIComponent(discussionId)}/notes`,
    { body },
  );
},
```

#### 4. Update `commitFiles` serialization

Handle the new action types when building the GitLab API body:

```typescript
async commitFiles(branch, actions, message) {
  await request('POST', '/repository/commits', {
    branch,
    commit_message: message,
    actions: actions.map((a) => {
      if (a.action === 'delete') {
        return { action: 'delete', file_path: a.filePath };
      }
      if (a.action === 'move') {
        return { action: 'move', file_path: a.filePath, previous_path: a.previousPath, content: a.content, encoding: 'text' };
      }
      return { action: a.action, file_path: a.filePath, content: a.content, encoding: 'text' };
    }),
  });
},
```

## Phase 4: Respond Core Logic

### Overview

New `src/gitlab/respond.ts` with `respondOne`. Reads the submissions file, fetches all MR discussions, processes tracked discussions and diff discussions, batches commits, posts reply confirmations.

### Changes Required

#### 1. `src/gitlab/respond.ts` (new file)

**Context type:**

```typescript
export interface RespondCtx {
  ledgerPath:     string;
  submissionsDir: string;
  proposalsDir:   string;
  contentDir:     string;
  transcriptsDir: string;
  clientFn?:      (baseUrl: string, token: string, projectId: string) => GitLabClient;
  completeFn?:    typeof defaultComplete;
}
```

**Top-level function — `respondOne(entry, ledger, ctx)`:**

1. Guard: `entry.stages.prOpened === null` → throw `"MR not opened — run submit first"`
2. Read `state/submissions/<basename>.json` → get `mrIid` + `discussions` mapping (throw if missing)
3. Build GitLab client
4. `client.listDiscussions(mrIid)` → all discussions
5. Build a `Set<string>` of tracked discussion IDs from the submissions file
6. **Process tracked discussions** (speculative/contradiction notes we posted):
   - For each `DiscussionMapping`, find the matching `Discussion` in the list
   - If discussion not found: skip (MR may have been modified externally)
   - Check for our sentinel replies: if any note body starts with `"Applied."` or `"Denied."` → skip (already processed)
   - Find the first non-first note whose body matches `/^approve\b/i` or `/^deny\b/i`
   - If "approve": call `applySpeculativeApproval(proposal, ctx)` → accumulate `CommitAction[]`; queue `"Applied."` reply on this discussion
   - If "deny": queue `"Denied — no changes applied."` reply; no commit
   - If neither: skip (no response yet)
7. **Process diff discussions** (inline comments on files):
   - Filter discussions to those where `notes[0].position !== null` and `id` not in tracked set
   - Skip if any note body === `"Applied."` (bot already handled)
   - Call `applyDiffComment(discussion, ctx)` → accumulate `CommitAction[]`; queue `"Applied."` reply
8. If any `CommitAction[]` accumulated: call `client.commitFiles(branch, allActions, message)`
   - Branch name: read from submissions file or re-derive as `wiki/<basename>`
   - Commit message: `wiki: apply MR responses for <basename>\n\n<N> speculative approval(s) applied\n<M> diff comment(s) applied`
9. Post all queued replies via `client.addDiscussionNote`

**`applySpeculativeApproval(proposal: CommentProposal, filename: string, ctx, completeFn): Promise<CommitAction[]>`**

- If `proposal.relatedPath === null`: return `[]` (no page to edit; log warning)
- Read `content/${proposal.relatedPath}` from disk (empty string if not found → treat as create)
- Read transcript lines at `proposal.citations` from `transcripts/<filename>`
- LLM call (same `complete` wrapper as propose stage):
  - System prompt (mirrors `PROPOSER_SYSTEM_TEMPLATE`):
    ```
    You incorporate a stakeholder-approved claim into a Pathfinder 2e campaign wiki page.
    The claim was initially flagged as speculative but has now been approved.
    Given the current page content and the approved claim, output exactly ONE proposal:
      - kind:"edit"   — literal-substring replacement (oldText must appear exactly once in the page)
      - kind:"append" — add content under a heading (afterHeading: null for end-of-file)
      - kind:"create" — new page (only if the page is empty/does not exist)
    Follow wiki conventions verbatim. Do NOT add information beyond what the claim states.
    --- WIKI CONVENTIONS ---
    {{CONVENTIONS}}
    ```
  - User message: current page content + claim text + transcript excerpt
  - Output schema: `z.object({ proposal: z.discriminatedUnion('kind', [EditSchema, AppendSchema, CreateSchema]) })`
- Convert the LLM proposal to `CommitAction[]` via `buildCommitActions([{ ...llmProposal, path: proposal.relatedPath, citations: proposal.citations }], { contentDir: ctx.contentDir })`

**`applyDiffComment(discussion: Discussion, ctx, completeFn): Promise<CommitAction[]>`**

- Extract `position.new_path` from `discussion.notes[0]` → the affected file path (repo-relative, e.g. `content/Org/Foo.md`)
- Extract the instruction from the first non-system note body
- Read current file content from disk at `ctx.contentDir/...` (strip `content/` prefix)
- LLM call:
  - System prompt:
    ```
    You apply a stakeholder's instruction to a wiki file.
    Instructions are short free-form requests such as:
      "rename to X.md", "delete this entry", "merge content into [[Other Page]]"
    Output a JSON array of CommitActions. Each element is one of:
      { "action": "update", "filePath": "<repo-relative path>", "content": "<full new file content>" }
      { "action": "delete", "filePath": "<repo-relative path>" }
      { "action": "move",   "filePath": "<new repo-relative path>", "previousPath": "<old path>", "content": "<full content if changed>" }
    Use the exact file path provided. Do NOT add, remove, or rename other files.
    ```
  - User message: instruction text + current file content + file path
  - Output schema: `z.object({ actions: z.array(CommitActionSchema) })` where `CommitActionSchema` is a discriminated union on `action`
- Return the validated actions array

**Branch name for the respond commit:**

Store `branch` in the submissions file during Phase 3 (add `branch: string` field to `SubmissionsFile`), so `respondOne` knows which branch to commit to without re-deriving it.

> Note: update `src/gitlab/submit.ts` in Phase 3 to also write `branch` into the submissions file.

## Performance Considerations

- `listDiscussions` paginates at 100 per page; most MRs have far fewer discussions than that.
- All commit actions accumulated across one respond run are batched into a single `commitFiles` call to avoid multiple commits cluttering the MR timeline.

## Migration Notes

Existing MRs opened before this change used flat `addNote` — they have no submissions file and no discussion IDs. `respond` will skip them gracefully (submissions file missing guard in Phase 4). Stakeholders would need to re-submit those transcripts to get threadable discussions; this is acceptable since the feature is new.
