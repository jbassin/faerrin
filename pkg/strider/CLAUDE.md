# CLAUDE.md

This file provides guidance to Claude Code when working in this repository.

## Project Overview

Strider is a statically-rendered interactive website displaying a map of "The Strider," a city in a Pathfinder 2e tabletop roleplaying campaign. The map shows 20 factions controlling territory across a circular hex grid; clicking a faction reveals its symbol, description, and known members.

The site is statically built (to `dist/client`) and served by a Caddy reverse proxy configured at the repo root in `sites.caddyfile` (the `strider.iridi.cc` block). That file is **gitignored** (it holds a Cloudflare DNS token), so it lives in the working tree on the host but is not version-controlled.

## Content & Build Pipeline

`scripts/build-content.ts` reads every `content/factions/*.md` and `content/layers/*.md`, converts each markdown body to HTML with `remark`, and emits typed TS modules under `src/generated/`. The runtime app imports these — **no `fs`, `remark`, or `gray-matter` reaches the client bundle**.

- The Vite plugin in `scripts/contentWatchPlugin.ts` regenerates these modules on every `bun run build` (via `buildStart`) and on every `content/**/*.md` change during `bun dev` (via the configureServer watcher).
- The Vitest global setup (`vitest.global-setup.ts`) regenerates them once if missing, so unit tests work after a fresh `git clone`.
- `src/lib/{factions,layers}.ts` are thin wrappers around the generated module — they preserve the original API (`getAllFactions`, `getFactionBySlug`, `getAllLayers`, `getCurrentRegions`) but the implementation is a constant-time lookup.

## Editor route (`/editor`)

`/editor` is **dev-only**. It exists when running `bun dev` and talks to `scripts/editor-server.ts` (bound to `0.0.0.0:3001`) to write new layer markdown files. Vite's `routeFileIgnorePattern: 'editor\\.tsx$'` (active when `NODE_ENV === 'production'`) strips the route file from `routeTree.gen.ts` during `bun run build`, so the production bundle contains no editor code and `/editor` returns 404 on the deployed site.

## Code Standards

### TypeScript

- `strict: true` is required. Do not relax it.
- Avoid `any`. If genuinely needed, leave a one-line comment explaining why.
- Use type-only imports where applicable: `import type { Faction } from '@/lib/factions'`.

### React Components

- TanStack Start has no Server Components — every component renders on both server (prerender) and client (hydration). Anything that touches the DOM, WebGL, or `window`/`document` directly (Pixi.js is the load-bearing example — the entire map renderer lives behind it) must be wrapped in `<ClientOnly>` and lazy-imported so its module never evaluates server-side.
- Component files use PascalCase: `HexMap.tsx`. CSS modules are co-located: `HexMap.module.css`.
- Keep components focused. If a file approaches ~200 lines, split it.

### File Organization

- Pure logic → `src/lib/` (testable without React).
- React components → `src/components/`.
- Routes → `src/routes/` (TanStack file-based routing; `$param` for dynamic segments, `__root.tsx` for the document shell).
- Avoid barrel/index re-exports; import from the source file directly.

### Formatting & Linting

- Prettier with default config. Do not add a `.prettierrc` with overrides.
- ESLint flat config with `typescript-eslint` recommended + `eslint-plugin-react-hooks`.

## Content Authoring

Faction files in `content/factions/{order:02d}-{slug}.md` follow this schema:

```yaml
---
name: string # Display name
color: string # CSS hex color
symbol: string|null # Path under public/, or null if no SVG yet
---
```

`order` (1–20) and `slug` are derived from the filename — do not add them to the frontmatter.

Body structure:

1. Description paragraphs (before any heading)
2. Optional `## Known Members` section with `### Member Name` subsections and bio prose below each

**Faction content is authored material.** Do not modify files in `content/factions/` unless the user explicitly asks. Schema, data-loading, and rendering changes are fine; content edits require user direction.

Layer files in `content/layers/{YYYY}-{MM}-{DD}T{HHMMSS}-{slug}.md` use chronological filenames so directory order matches event order without manual numbering. Schema is `{ timestamp, message, changes: Array<Change> }` where `Change` covers region mutations (`add`/`update`/`remove`), territory ownership (`claim`), and the skein overlay graph (`skein-add`/`skein-update`/`skein-remove`/`skein-connect`/`skein-disconnect`) — see `src/lib/regions.ts` for the full discriminated union and `content/layers/CLAUDE.md` for examples.

## Systems Overview

- **Hex grid model** (`src/lib/hexUtils.ts`) — Axial coordinates `(q, r)` on a flat-top layout, radius-35 grid. 20 faction territories are arranged as a clock around a Harlequins-owned donut hole; hex-to-faction assignments and border edge sets are precomputed at module load.
- **Layer / Change pipeline** — `scripts/build-content.ts` parses each `content/layers/*.md`, validates every `Change`, then emits `src/generated/layers.ts` containing both the raw `LAYERS` array and pre-folded `CURRENT_*` snapshots (regions, skein, per-faction hexes, borders). `foldRegions` / `foldSkein` / `foldFactionOverrides` in `src/lib/regions.ts` are the pure reducers used by both the build and any runtime that needs to fold a sub-sequence of layers.
- **Skein system** — A separate overlay graph rendered on top of region state. Skein regions (`skein-add`/`-update`/`-remove`) are single-hex nodes with their own symbol; `skein-connect`/`skein-disconnect` form an undirected edge set drawn as bowed curves with pulsing comets along them. Layout/geometry lives in `src/components/HexMap/skeinGeometry.ts`.
- **Animation orchestration** — `src/components/MapView/useTimelinePlayback.ts` owns timeline cursor + autoplay; `useTypewriter.ts` drives the per-layer message reveal. The Pixi-side `src/components/HexMap/animationManager.ts` consumes a `LayerAnimation` hint (region adds, skein connects, faction-flip targets) emitted when the timeline advances by exactly one step and stages the visual effects.

## Testing

- **Unit tests** live alongside the code as `*.test.ts` (Vitest auto-discovers them).
- **E2E tests** live in `e2e/*.spec.ts` (Playwright). They build via `bun run build` and serve `dist/client` on port 3000.
- Pure logic in `src/lib/` should be unit-tested. Components don't require unit tests unless they contain non-trivial logic.
- E2E tests must cover the responsive flow: desktop modal open/close, mobile faction-page navigation.

## Commit Conventions

This project uses **Conventional Commits**:

```
<type>(<scope>): <description>

[optional body]
```

Types: `feat`, `fix`, `chore`, `docs`, `refactor`, `test`, `style`, `perf`, `build`, `ci`.

Examples:

- `feat(map): add hex-to-faction assignment`
- `fix(detail): close modal on Escape key`
- `chore: bump pixi.js to 8.x`

## Ticket Workflow

Tickets live in `tickets/` as markdown files — this project does not use Linear or any external tracker. To pick up a ticket, read its file and `tickets/README.md` for context. When a ticket is complete, check off its success criteria in the file.
